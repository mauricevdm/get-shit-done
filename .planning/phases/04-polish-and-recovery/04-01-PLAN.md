  <name>Task 1: Add cmdHealthCheck with comprehensive detection</name>
  <files>get-shit-done/bin/gsd-tools.cjs</files>
  <action>
Add new function `cmdHealthCheck(cwd, options, raw)` that combines worktree health detection:

1. **Orphan detection (RECV-01):**
   - Call existing `cmdWorktreeStatus` internally to get discrepancies
   - Map discrepancy types to issue objects:
     - `path_missing` -> orphan (registry points to deleted path)
     - `not_in_git` -> orphan (path exists but git doesn't track it)
     - `not_in_registry` -> orphan (git worktree not in our registry)
   - For each registry entry with status='active', calculate age from `created` timestamp
   - If age > threshold (default 7 days, configurable via `options.ageThreshold`), add `age_exceeded` issue

2. **Stale lock detection (RECV-02):**
   - Iterate registry.locks entries
   - For each lock, use existing stale detection logic (kill -0 PID check)
   - Add `stale_lock` issue for dead PIDs
   - Include hostname in issue metadata (per RESEARCH.md pitfall #3: warn if different host)

3. **Incomplete finalization detection (RECV-03):**
   - Check for marker files in `.planning/worktrees/finalization/phase-*.json`
   - For each marker file found:
     - Parse JSON to get `steps` status
     - Add `incomplete_finalization` issue with details of which steps completed
   - Also check git state: if `.git/MERGE_HEAD` exists, add `merge_in_progress` issue

4. **Exit code computation (per CONTEXT.md):**
   ```javascript
   const EXIT_CODES = {
     HEALTHY: 0,
     ORPHANS_ONLY: 1,
     INCOMPLETE_ONLY: 2,
     ORPHANS_AND_INCOMPLETE: 3,
     RUNTIME_ERROR: 4
   };
   ```
   - Return `exit_code` in output based on issue types found

5. **Output structure:**
   ```javascript
   {
     status: 'healthy' | 'degraded' | 'broken',
     issues: [
       { type, phase, path, branch, age_days, suggested_action, repairable, metadata }
     ],
     exit_code: number,
     summary: { orphan_count, stale_lock_count, incomplete_count }
   }
   ```

Add CLI routing: `health check [--age-threshold N]` (days)

Follow existing patterns from `cmdValidateHealth` for structure.

Update finalize-phase.md to write marker files before cleanup operations:

1. **Before cleanup_worktree step**, add new step `write_finalization_marker`:
   ```markdown
   <step name="write_finalization_marker">
   Create marker file to track finalization progress for recovery:

   ```bash
   MARKER_DIR=".planning/worktrees/finalization"
   MARKER_FILE="$MARKER_DIR/phase-${PHASE}.json"
   mkdir -p "$MARKER_DIR"

   cat > "$MARKER_FILE" << EOF
   {
     "phase": ${PHASE},
     "worktree_path": "${WORKTREE_PATH}",
     "branch": "${BRANCH}",
     "started": "$(date -u +%Y-%m-%dT%H:%M:%SZ)",
     "steps": {
       "state_reconciled": true,
       "merged": true,
       "worktree_removed": "pending",
       "lock_released": "pending",
       "registry_updated": "pending"
     }
   }
   EOF
   ```

   This marker enables `/gsd:health` to detect and recover from interrupted finalization.
   </step>
   ```

2. **Update cleanup_worktree step** to update marker after worktree removal:
   - After successful `phase-worktree.sh remove`, update marker: `worktree_removed: true`
   - After lock release, update marker: `lock_released: true`
   - After registry update, update marker: `registry_updated: true`

3. **At end of workflow**, add step `remove_finalization_marker`:
   ```markdown

   MARKER_FILE=".planning/worktrees/finalization/phase-${PHASE}.json"
   rm -f "$MARKER_FILE"
   ```

   Marker removal indicates clean completion. If marker remains, `/gsd:health` can recover.
   </step>
   ```
